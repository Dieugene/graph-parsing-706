# Декомпозиция задач

## Отправная точка: одна задача

Начинай анализ с вопроса: **можно ли реализовать всё одной задачей?**

Analyst имеет право дробить в пределах своей компетенции. Твоя задача — не создавать избыточных границ.

## Когда дробить

Дроби только если одной задачей нельзя:
- Независимые модули (параллельная работа)
- Разные технологические домены
- Экспериментальные части (могут не пригодиться)

## После дробления: ревизия

Проверь возможность объединения:

**Однородность:** задачи одного типа (работа с API, работа с файлами) лучше объединить.

**Стыковка:** модули в одном workflow лучше в одной задаче — сразу тестируется "бесшовность".

## Границы и контракты

При дроблении внутри каждой задачи накапливается ошибка. Границы и контракты — точки сверки для агентов.

**В task_brief фиксируй:**
- Входные/выходные интерфейсы задачи
- Контракты взаимодействия с соседними модулями
- Критерии проверки на границах

## Сверка с существующим кодом

**Перед началом:**
1. Проверь доступ к проектам-источникам (указаны в overview.md)
2. Нет доступа → запроси у пользователя
3. Изучи паттерны, переиспользуй
